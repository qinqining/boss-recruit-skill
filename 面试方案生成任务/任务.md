# 面试方案生成任务

执行本任务前阅读 **`岗位层级说明.md`**。素质表为 **SEO 运营档**，默认面试岗位为 **SEO 助理**：须落实助理向校准（文首声明、分档与题干降维），见该文件。

## 输入

| 类型 | 路径 |
|------|------|
| 岗位 JD 与门槛 | `面试方案生成任务/岗位素质项/01_岗位与JD门槛.md` |
| 素质分档与面试母题（含高优先级母题标注） | `面试方案生成任务/岗位素质项/02_素质分档与面试母题.md` |
| 层级与录用线口径 | `面试方案生成任务/岗位层级说明.md` |
| 候选人材料 | 默认 `面试方案生成任务/候选人简历/`。**PDF 不利于 Agent 精读**：先抽成 Markdown 再读——`pip install pymupdf` 后执行 `py scripts/pdf_resume_to_md.py <路径>.pdf`（默认同名 `.md`）。扫描件需 OCR 或人工粘贴。或由用户给出已整理的正文/Markdown 路径。 |
| 已处理简历（归档） | `面试方案生成任务/已处理简历/`。仅在**用户明确确认**面试方案定稿后，将本次对应候选人的简历文件从 `候选人简历/` **移动**至此目录（见下文「用户确认后」）。 |
| 实际门槛（若存在） | `面试方案生成任务/SEO助理岗位要求.md`；存在则以该文件为准，上表 JD 章节仅作母题与维度参考 |

**母题覆盖**：`02_素质分档与面试母题.md` 中标注为「**高优先级母题**」的段落，生成方案时须覆盖其考察意图；题干须结合候选人材料个性化改写，不得无故删改考察意图。

## 岗位层级（执行摘要）

| 项 | 规则 |
|----|------|
| 录用线 | 助理主参照 **4–6 分**；**7–10 分** 为加分追问，不作默认必达门槛 |
| 母题 | 保留高优先级母题意图，表述降为 **助理语境**（执行、学习、协作、在指导下闭环） |
| JD | 若 `01_…` 中职责为运营全责描述，方案中须明确 **已收缩为助理职责**；用语以协助/参与/在监督下完成为主 |

## 约束

- 不得编造简历中不存在的经历、数据、项目名；追问须可追溯到候选人材料或自述。
- **统筹规划不作为主要考察素质项**（助理岗位侧重执行配合，而非独立统筹多线任务），Q4/Q5 若引用跨行业经历（如学生会、土建施工），须改为**候选人本岗位 SEO 相关经历**提问。
- 问题应多围绕候选人**SEO 相关工作经历**展开（关键词研究、内容优化、数据监控、技术 SEO 等），减少跨行业类比。

## 输出

1. **主问题 10 条**（每条下可有 1–2 句建议追问，不计入 10 条）。
2. 每条须含：**考察素质项**（对应 `02_…` 中某一 `##` 节）、**简历锚点**（一句话）。
3. 文首：**方案日期**、对外岗位名；并含助理校准声明（语义与 `岗位层级说明.md` 中模板一致）。
4. **交付物（必须）**：在 `面试方案生成任务/输出/` 写入 **Microsoft Word `.docx`**，命名与日期规则同下；**`.md` 可作为中间稿或附件，不得替代 `.docx` 作为唯一交付**。
   - Word：`面试方案_YYYY-MM-DD_<候选人标识>.docx`；同日多版：`面试方案_YYYY-MM-DD_HHmm_<候选人标识>.docx`
   - 可选中间稿：`面试方案_YYYY-MM-DD_<候选人标识>.md`（便于 diff；定稿仍以 `.docx` 为准）
5. **生成 Word**：在已定稿的 Markdown 上执行（仓库根目录）  
   `pip install python-docx`  
   `py scripts/interview_plan_md_to_docx.py 面试方案生成任务/输出/<同名>.md`  
   默认同路径生成同名 `.docx`。若本机已安装 **Pandoc**，也可改用 `pandoc -f gfm -o …docx …md` 等等价方式，以版式更接近原文为准。

（3–6 条）

## 自检

- 高优先级母题均已体现且已助理向改写  
- 文首含日期与助理校准声明  
- 录用线未按 9–10 分运营档误设  
- 无简历外编造  
- **`输出/` 目录下已存在与方案对应的 `.docx`**
- 若用户已确认定稿：对应候选人的简历文件已从 `候选人简历/` 移至 `已处理简历/`

## 用户确认后（归档简历）

当用户**明确确认**面试方案可定稿（例如：「方案确认」「定稿了」「可以归档简历」「面试方案 OK」等）时，Agent **须执行**：

1. **确定对应文件**：根据本轮交付的 `面试方案生成任务/输出/面试方案_*_<候选人标识>.docx`（或同名 `.md`）中的 **候选人标识**，在 `面试方案生成任务/候选人简历/` 下匹配同一候选人的材料（常见为同名或前缀一致的 **`.pdf`、`.md`**；若仅 PDF 无 md，只移 PDF；若两者皆有则一并移动）。
2. **移动（非复制）**：将上述匹配到的文件从 `候选人简历/` **移动**到 `面试方案生成任务/已处理简历/`（使用 PowerShell `Move-Item` 或 Python `shutil.move`，避免重复归档导致两份并存）。
3. **冲突处理**：若 `已处理简历/` 下已存在同名文件，与用户确认是否覆盖，或改为带时间后缀的文件名后再移动。
4. **未确认前**：不得擅自移动 `候选人简历/` 内文件；用户仅要求「生成方案」而未确认时，材料保留在 `候选人简历/` 供修改后再版。

## 隐私

输出含评价倾向；`输出/`、`已处理简历/` 默认不入库。引用简历宜最短必要。
